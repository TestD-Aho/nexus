Ce document constitue une analyse "pro" de l'intégralité de la logique contenue dans le dossier @app/.

======================================================================================================================================================
PARTIE I : LES FONDATIONS TECHNIQUES (CORE & CONFIG)
======================================================================================================================================================

1. app/Core/Bootstrap.php
-------------------------
Le point d'entrée technique. Sa logique est centrée sur la configuration de l'environnement d'exécution.
- Fonctionnement : Il initialise les paramètres PHP critiques (ini_set).
- Sécurité : Désactive display_errors pour éviter les fuites d'informations sensibles en production.
- Monitoring : Configure un log d'erreurs centralisé dans logs/php-error.log, permettant une maintenance proactive.
- Life of app : Sans lui, l'application pourrait exposer des vulnérabilités lors d'erreurs SQL ou de chemins.

2. app/Core/Autoload.php
------------------------
Le chef d'orchestre du chargement des classes.
- Logique : Implémente un système hybride. Il commence par charger vendor/autoload.php (Composer) pour les bibliothèques tierces.
- PSR-4 Fallback : Définit une fonction spl_autoload_register pour le namespace "CheckMaster\".
- Transformation : Convertit les backslashes du namespace en DIRECTORY_SEPARATOR pour localiser les fichiers .php dans app/.
- Importance : Assure que chaque composant de app/ (Services, Models, etc.) est disponible à la demande sans inclusion manuelle.

3. app/Core/Router.php
----------------------
Le moteur de navigation de la nouvelle architecture.
- Logique : Stocke une liste de routes avec leur méthode HTTP, leur pattern et leur handler.
- Dispatching : Analyse la requête entrante, compare l'URI aux patterns enregistrés (incluant les paramètres dynamiques).
- Response : Garantit que chaque handler renvoie un objet CheckMaster\Core\Response.
- Statut : C'est le pivot de la transition vers un MVC moderne.

4. app/Core/Request.php & Response.php
--------------------------------------
Abstraction des flux HTTP.
- Request : Encapsule $_GET, $_POST, $_SERVER, et $_FILES. Offre une méthode fromGlobals() pour normaliser l'entrée.
- Response : Gère les codes de statut HTTP (200, 404, 403, etc.), les en-têtes (Content-Type) et le corps de la réponse.
- Intégrité : Empêche l'utilisation sauvage des superglobales dans les services modernes.

5. app/Core/Session.php & Csrf.php
----------------------------------
Couche de persistance et de protection contre les attaques transverses.
- Session : Durcit la gestion des sessions (use_only_cookies, httponly, samesite=Lax). Implémente regenerate() pour éviter la fixation de session.
- Csrf : Génère et valide des jetons de sécurité. Sa méthode injectIntoPostForms() est cruciale dans layout.php pour protéger les formulaires legacy.

6. app/config/database.php
--------------------------
Le point de connexion unique.
- Logique : Utilise le pattern Singleton pour fournir une instance PDO unique.
- Paramétrage : Définit l'encodage UTF8MB4, le mode d'erreur (Exception) et le mode de fetch par défaut (ASSOC).
- Alignement : C'est le lien direct avec la base de données décrite dans audit.txt.

7. app/config/permission_registry.php
-------------------------------------
Le "cerveau" du RBAC (Role-Based Access Control).
- Logique : Définit une structure de données massive associant chaque fonctionnalité à des droits (voir, creer, modifier, supprimer).
- Groupes : Mappe les rôles (Admin, Secrétaire, Étudiant) aux IDs de la table groupe_utilisateur.
- Routes : Mappe les URLs aux permissions requises. C'est ici que l'on définit si une page est "publique" ou "privée".

PARTIE II : LA COUCHE DE SÉCURITÉ (SECURITY & MIDDLEWARES)
======================================================================================================================================================

8. app/middlewares/PermissionMiddleware.php
-------------------------------------------
Le garde-barrière des requêtes.
- Logique : Intercepte chaque requête avant qu'elle n'atteigne le contrôleur.
- Détection d'Action : Analyse $_GET['action'] pour déterminer dynamiquement si l'utilisateur tente une lecture ou une écriture.
- Audit : Si l'accès est refusé, il utilise AuditLog pour enregistrer l'incident (table pister).
- Intégration : Indispensable pour maintenir la sécurité sur les routes legacy qui n'utilisent pas encore le Router moderne.

9. app/Security/AuthorizationService.php
----------------------------------------
Le moteur de décision des droits.
- Fonctionnement : Reçoit une demande d'accès pour un utilisateur, une fonctionnalité et une action.
- Logique SQL : Interroge les tables fonctionnalite et permission pour vérifier si le groupe de l'utilisateur possède le flag requis.
- Optimisation : Utilise un cache statique ($resolveCache) pour ne pas ré-interroger la DB plusieurs fois par seconde pour le même droit.

10. app/Security/RouteActionResolver.php
----------------------------------------
Analyseur d'intention.
- Rôle : Transforme une URI ou un paramètre de page en une action sémantique (ex: "modifier_etudiant" -> action "modifier" sur fonctionnalité "gestion_etudiant").
- Précision : Permet au système de sécurité de comprendre exactement ce que l'utilisateur essaie de faire.

======================================================================================================================================================
PARTIE VI : LE PARCOURS UTILISATEUR COMPLET (USER JOURNEY)
======================================================================================================================================================

Pour illustrer comment tout est branché, suivons le parcours d'un étudiant fictif :

ÉTAPE 1 : L'INSCRIPTION
- L'étudiant se présente à la scolarité.
- Le Secrétaire accède à public/layout.php?page=gestion_scolarite.
- InscriptionController / GestionScolariteController charge les frais via Scolarite::getMontantScolarite().
- Un versement est enregistré. Scolarite::creerInscription() insère une ligne dans inscriptions.
- AuditLog enregistre l'action.
- Résultat : L'étudiant est "Inscrit" (id_annee_acad et id_niveau sont fixés).

ÉTAPE 2 : LE STAGE
- L'étudiant trouve un stage et saisit ses infos dans CandidatureSoutenanceController.
- Le service vérifie que le stage durera bien 6 mois.
- La table informations_stage est alimentée. L'entreprise est liée (entreprises).

ÉTAPE 3 : LA CANDIDATURE
- Une fois le stage fini, l'étudiant clique sur "Soumettre Candidature".
- CandidatureSoutenanceService vérifie que l'étudiant est à jour de ses frais (reste_a_payer <= 0 ?).
- Une ligne est créée dans candidature_soutenance avec le statut 'En attente'.

ÉTAPE 4 : L'ÉVALUATION DU RAPPORT
- L'étudiant dépose son rapport PDF (GestionRapportController).
- La Commission (EvaluationDossiersController) reçoit le dossier.
- Chaque membre vote via EvaluationDossiersService.
- Une fois le consensus atteint, le rapport passe au statut 'valide'.

ÉTAPE 5 : LA PROGRAMMATION
- L'administrateur (ProgrammationSoutenanceController) voit l'étudiant dans la liste des "Validés".
- Il lui affecte une salle, une date et un jury (Président + Examinateurs).
- Une ligne est créée dans programmer_soutenance.

ÉTAPE 6 : LA SOUTENANCE ET LE DIPLÔME
- Le jour J, le jury saisit les notes dans EvaluationSoutenanceController.
- Le service calcule la moyenne finale pondérée par les coefficients de bareme_critere.
- Si Note >= 10, la soutenance est validée.
- Le PV Final est généré en PDF. L'étudiant a fini son parcours.

PARTIE VIII : LOGIQUE ALGORITHMIQUE DÉTAILLÉE
======================================================================================================================================================

28. Calcul du Reste à Payer (Scolarite.php)
-------------------------------------------
Algorithme :
1. Récupérer l'ID du niveau de l'étudiant.
2. Récupérer le montant fixe dans frais_inscription pour ce niveau et l'année académique.
3. Sommer les montants_verser dans la table inscriptions pour cet étudiant et cette année.
4. Reste à Payer = Montant Fixe - Somme des Versements.
5. Si Reste <= 0, marquer comme "Soldé".

29. Validation du Semestre (Etudiant.php)
-----------------------------------------
Algorithme :
1. Récupérer toutes les notes (notes) liées aux unités d'enseignement (ue) du semestre.
2. Séparer les notes en deux piles : Majeures (credits > 3) et Mineures (credits <= 3).
3. Calculer Moyenne_Majeure = Somme(Notes_Majeures) / Nombre(UE_Majeures).
4. Calculer Moyenne_Mineure = Somme(Notes_Mineures) / Nombre(UE_Mineures).
5. Semestre_Valide = (Moyenne_Majeure >= 10 ET Moyenne_Mineure >= 10).
Note : Si l'une des deux moyennes est < 10, le semestre n'est pas validé, même si la moyenne globale est > 10.

30. Gestion des Conflits de Planning (ProgrammationSoutenanceService.php)
-------------------------------------------------------------------------
Algorithme :
1. Avant d'insérer dans programmer_soutenance :
2. Vérifier si l'enseignant X est déjà affecté à un autre jury à la même date et heure.
3. Vérifier si la salle S est déjà occupée au même moment.
4. Vérifier si l'étudiant n'a pas déjà une autre soutenance programmée.
5. Si un conflit est détecté, renvoyer une erreur JSON détaillée.

PARTIE XII : ANALYSE DES FLUX DE DONNÉES ET DES BRANCHEMENTS
======================================================================================================================================================

La "vie" de @app/ est un cycle continu de validation et de transformation de la donnée.

Branchement 1 : Authentification -> Session -> RBAC
Le contrôleur AuthController utilise le service AuthService qui lui-même interroge le modèle Utilisateur. 
Une fois validé, l'ID de session est utilisé par PermissionMiddleware pour consulter PermissionRegistry. 
C'est le branchement fondamental qui sécurise tout l'accès à app/.

Branchement 2 : Inscription -> Finance -> Statut Étudiant
Le Secrétaire passe par GestionScolariteController. Le service GestionScolariteService utilise Scolarite.php 
pour mettre à jour inscriptions. Si le paiement est complet, le modèle Etudiant.php est mis à jour 
automatiquement pour refléter le nouveau statut.

Branchement 3 : Stage -> Candidature -> Commission
L'étudiant remplit ses infos via CandidatureSoutenanceController. CandidatureSoutenanceService vérifie 
la règle des 6 mois. Une fois validé, le dossier apparaît dans EvaluationDossiersController pour les enseignants.

Branchement 4 : Rapport -> Jury -> Note Final
Le rapport est déposé via GestionRapportController. ProgrammationSoutenanceController lui affecte un Jury. 
Le jour de la soutenance, EvaluationSoutenanceController enregistre les notes dans evaluer. 
Le service PvFinalGeneratorService clôture la boucle en produisant le PV PDF.

PARTIE XIII : ANALYSE DES REGRÈGLES MÉTIER CRITIQUES (DEEP DIVE)
------------------------------------------------------------------------------------------------------------------------------------------------------

Dans cette section, nous analysons les algorithmes les plus sensibles qui garantissent la validité académique.

Algorithme A : Vérification de la disponibilité des Jurys
L'application doit s'assurer qu'un enseignant ne soutient pas dans deux salles à la fois.
Logique :
1. Déterminer la plage horaire (H_début + 1h30).
2. Rechercher dans programmer_soutenance toutes les attributions pour cet enseignant à cette date.
3. Vérifier l'intersection des intervalles [H_début, H_fin].
4. Si Intersection != vide, renvoyer ERREUR_CONFLIT_HORAIRE.

Algorithme B : Calcul du Résultat Final de Soutenance
Ce n'est pas une simple moyenne.
Logique :
1. Récupérer les notes de chaque critère (C1, C2, C3...).
2. Récupérer les coefficients (K1, K2, K3...) de bareme_critere pour l'année académique.
3. Calculer Note_Brute = (C1*K1 + C2*K2 + ...) / Somme(K).
4. Si Note_Brute >= 10, Statut = "Admis".
5. Si Note_Brute < 10, Statut = "Ajourné".
6. La mention (Passable, Assez Bien, Bien, Très Bien) est calculée dynamiquement sur la Note_Brute.

Algorithme C : Gestion de l'Année Académique Glissante
Comment le système sait-il quelle année est "active" ?
Logique :
1. Par défaut, utilise l'année définie dans app_settings comme "annee_courante".
2. Si l'administrateur change l'année dans le sélecteur, la session est mise à jour via AcademicYear::setSelectedId().
3. Toutes les requêtes SQL subséquentes (Scolarité, Candidatures, Notes) incluent automatiquement une clause "WHERE id_annee_acad = :session_id".
4. Cela permet de consulter les archives tout en travaillant sur le présent.

PARTIE XVII : ANALYSE DU SYSTÈME RBAC EN PRATIQUE
------------------------------------------------------------------------------------------------------------------------------------------------------

Le système est "Permission-Driven". 
Exemple de flux de décision :
1. L'utilisateur clique sur "Supprimer un étudiant".
2. L'URL est layout.php?page=gestion_etudiant&action=supprimer.
3. PermissionMiddleware détecte l'action 'supprimer' sur la fonctionnalité 'gestion_etudiant'.
4. AuthorizationService regarde dans permission_registry.php.
5. Il trouve que pour le rôle 'secretaire', le flag 'peut_supprimer' pour 'gestion_etudiant' est FALSE (seul l'admin peut supprimer).
6. Le middleware intercepte et redirige vers access_denied.php.
Cette logique garantit l'intégrité des données face aux erreurs de manipulation ou aux tentatives d'intrusion.

PARTIE XVIII : ANALYSE TECHNIQUE ULTRA-DÉTAILLÉE DU SYSTÈME D'AUTORISATION
------------------------------------------------------------------------------------------------------------------------------------------------------

Le système d'autorisation de CheckMaster est une prouesse de conception hybride. Il doit faire cohabiter un routage moderne (basé sur des objets) et un routage legacy (basé sur des paramètres d'URL).

A. Le Registre des Permissions (permission_registry.php)
Ce fichier n'est pas qu'un simple tableau ; c'est un graphe de dépendances. 
Chaque entrée 'features' contient :
- code : Le code unique utilisé en base de données.
- slug : L'identifiant textuel utilisé dans le code (ex: 'gestion_etudiant').
- label : Le nom lisible dans le menu.
- category : Le regroupement logique (SCOLARITE, SOUTENANCE, etc.).
- menu_url : Le lien direct pour le MenuController.
- routes : Un tableau de patterns (ex: '/etudiants', '/etudiants/creer') avec les méthodes HTTP autorisées.
- permissions : Un mapping fin des actions. Par exemple, l'action 'creer' sur la route POST /etudiants nécessite la capacité 'creer' sur la fonctionnalité 'gestion_etudiant'.

B. Le RouteActionResolver : Le décodeur d'intention
Sa logique interne est la suivante :
1. Récupérer l'URI de la requête.
2. Parcourir toutes les fonctionnalités du registre.
3. Pour chaque fonctionnalité, parcourir ses routes.
4. Si l'URI matche le pattern et que la méthode HTTP correspond, renvoyer un objet PermissionContext.
Ce contexte contient l'ID de la fonctionnalité et l'action sémantique (voir, creer, modifier, supprimer). 
Cela transforme une URL brute en une intention métier claire.

C. L'AuthorizationService : Le juge de paix
Il implémente la méthode isAuthorized(id_utilisateur, fonctionnalite_slug, action).
Logique SQL détaillée :
"SELECT p.peut_voir, p.peut_creer, p.peut_modifier, p.peut_supprimer 
 FROM permission p
 INNER JOIN fonctionnalite f ON p.id_fonctionnalite = f.id_fonctionnalite
 INNER JOIN utilisateur u ON u.id_groupe_utilisateur = p.id_groupe_utilisateur
 WHERE u.id_utilisateur = :uid AND f.code_fonctionnalite = :code"
Le service traduit ensuite l'action demandée (ex: 'creer') en le flag correspondant (p.peut_creer).

D. Le LegacyActionResolver : Le pont vers le passé
Pour les URLs comme layout.php?page=gestion_etudiant&action=modifier, ce service intervient pour "traduire" les paramètres GET en contexte de permission. Il assure que même le code vieux de 10 ans bénéficie de la protection RBAC moderne.

PARTIE XXIII : ANALYSE TECHNIQUE DU SUPPORT DE DONNÉES ET DES TRANSACTIONS
------------------------------------------------------------------------------------------------------------------------------------------------------

La robustesse du système repose sur la classe app/Support/Database.php.

A. Gestion des Transactions PDO
Pour des opérations critiques comme l'inscription (qui touche à inscriptions et etudiants), le système utilise des transactions.
Logique :
1. $db->beginTransaction() : Verrouille les tables concernées.
2. Exécution de l'insertion financière.
3. Exécution de la mise à jour du profil étudiant.
4. $db->commit() : Valide les deux opérations.
5. En cas d'erreur, un catch() déclenche $db->rollBack(), assurant qu'on n'a pas un étudiant payé mais non inscrit.

B. Singleton et Connexion Persistante
La classe Support\Database garantit qu'une seule connexion est ouverte par requête, optimisant l'usage des ressources du serveur MySQL de l'UFHB. Elle configure également le mode d'erreur PDO::ERRMODE_EXCEPTION, forçant le développeur à gérer proprement chaque cas d'échec SQL.

PARTIE XXVI : STANDARDS DE SÉCURITÉ ET PROTECTION DES DONNÉES
------------------------------------------------------------------------------------------------------------------------------------------------------

CheckMaster applique des standards de sécurité de niveau "entreprise".

A. Hachage des Mots de Passe
L'application n'utilise jamais de texte clair. Elle s'appuie sur l'algorithme PASSWORD_BCRYPT (via password_hash()), qui inclut un sel (salt) automatique pour chaque utilisateur, rendant les attaques par tables arc-en-ciel inefficaces.

B. Prévention de la Force Brute (DbRateLimiter)
Le service DbRateLimiter interroge la table auth_rate_limits. 
Logique :
1. Chaque échec de login incrémente un compteur pour le couple (Login, IP).
2. Au bout de 5 tentatives en moins de 10 minutes, le compte est temporairement bloqué.
3. Le déblocage est automatique après écoulement du délai de refroidissement.

C. Injection SQL
Toutes les requêtes dans app/models/ utilisent des requêtes préparées (prepared statements) avec des marqueurs de substitution (:param). Aucune variable utilisateur n'est concaténée directement dans le SQL.

PARTIE XXVII : GESTION DES ERREURS ET JOURNALISATION DES INCIDENTS (LOGGING)

La fiabilité de CheckMaster repose sur une gestion rigoureuse des exceptions.

A. La hiérarchie des erreurs :
- Erreurs Fatales : Interceptées par Bootstrap.php et redirigées vers logs/php-error.log.
- Exceptions Métier : Lancées par les Services (ex: InsufficientFundsException) et rattrapées par les Contrôleurs pour afficher un message utilisateur amical.
- Erreurs Silencieuses : Les échecs de connexion à des services tiers (ex: SMTP) sont loggués mais ne bloquent pas l'expérience utilisateur principale.

B. Centralisation des logs :
L'application utilise un format de log standard : [DATE] [NIVEAU] [UTILISATEUR_ID] MESSAGE. Cela permet d'utiliser des outils de monitoring externes pour analyser la santé du système en temps réel.
