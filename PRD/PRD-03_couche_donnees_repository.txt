# PRD-03 - Couche donnees: Repository, QueryBuilder, ORM mapper

## Metadata
- Version: 1.0
- Date: 2026-03-14
- Dependances amont: PRD-01, PRD-02
- Portee: data access backend

## Resume executif
Ce PRD introduit une couche donnees propre pour supprimer la duplication SQL, fiabiliser les acces DB et rendre les services metier lisibles. La couche cible combine Repository, QueryBuilder et mapper ORM leger, en gardant le schema existant strictement intact.

## Objectifs
1. Centraliser l'acces donnees par aggregate metier.
2. Retirer le SQL direct des controllers/services lourds.
3. Standardiser les requetes complexes et la pagination.
4. Gerer proprement les cles composites (cas `inscriptions`).

## Contraintes non negociables
1. Aucun changement de schema BDD.
2. Toutes les colonnes/tables doivent correspondre a `docs3\audit.txt`.
3. Interdiction des requetes concatenees non preparees.
4. Aucun fallback silencieux en cas d'erreur SQL.

## Probleme actuel observe
- SQL disperse dans de nombreux fichiers.
- Requetes similaires implementees plusieurs fois.
- Incoherences de conventions (noms champs legacy/courants).
- Difficultes de maintenance de requetes complexes (dashboards, archives, soutenances).

## Architecture cible de la couche data
```text
Service metier
  -> Repository
    -> QueryBuilder
      -> QueryExecutor (PDO + erreurs + logs)
        -> Mapper ORM leger (Entity <-> Row)
```

## Composants a creer
- `app\Support\Sql\QueryBuilder.php`

- `app\Support\Sql\QueryCatalog.php`

- `app\Repositories\ArchiveRepository.php`

## Regles de conception QueryBuilder
- Jointures explicites et whitelistees.
- Pagination uniforme (limit, offset, total).
- Tri whiteliste uniquement.
- Construction de requetes composees reutilisables.
- Binding parametres obligatoire.

## Regles ORM mapper leger
- Pas de magie runtime lourde.
- Mapping explicite entite <-> colonnes.
- Support des cles composites:
  - `inscriptions`: `num_carte_etud + id_annee_acad + num_versement`
- Modeles minces: pas de logique metier complexe.

### Frontieres transactionnelles obligatoires
- Transaction obligatoire:
  - enregistrement versement + audit
  - depot rapport + trace deposer + audit
  - decision commission + mise a jour statut + audit
  - evaluation soutenance + calcul note + enregistrement
- Interdiction de transaction geante cross-use-case.

### Regles SQL obligatoires
1. Requetes preparees uniquement.
2. Champs whitelistes pour ORDER BY.
3. Pagination native obligatoire sur listes volumineuses.
4. Aucun `SELECT *` en dehors des cas explicitement justifies.
5. Aucun alias table/colonne hors mapping documente.

### Mapping sensible a surveiller
- `inscriptions` : cle composite `num_carte_etud + id_annee_acad + num_versement`.
- Flux recu/paiement: ne pas reintroduire une dependance a `id_inscription`.
- `rapport_etudiants`: respecter les noms de colonnes reels (audit).
