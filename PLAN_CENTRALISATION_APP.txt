================================================================================
2) OBJECTIF REEL: REDUIRE LE CODE SANS CASSER LE METIER
================================================================================

Vous ne devez pas "mettre tout dans un seul fichier PHP".
Vous devez centraliser la logique dupliquee dans quelques composants pivots.

Objectif pragmatique:
- Moins de SQL duplique.
- Moins de controllers "copier-coller" par action.
- Moins de services "god class".
- Models plus minces, axes sur acces donnees coherent.
- Gestion explicite des alias legacy vs schema actuel.

================================================================================
3) FICHIERS PIVOTS A CREER (CENTRALISATION)
================================================================================

3.1 Couche SQL commune
Creer:
- app/Support/Sql/QueryCatalog.php
- app/Support/Sql/QueryExecutor.php
- app/Support/Sql/PaginationSql.php
- app/Support/Sql/SchemaAliasResolver.php

Role:
- QueryCatalog.php:
  stocke les requetes recurrentes (lecture etudiants, inscriptions, annee, rapport).
- QueryExecutor.php:
  execute prepare/execute/fetch avec conventions uniques (logs + erreurs + timing).
- PaginationSql.php:
  centralise LIMIT/OFFSET/ORDER BY whiteliste.
- SchemaAliasResolver.php:
  mappe les noms legacy -> schema courant (etudiant=>etudiants, etc).

Impact:
- suppression massive des SQL "quasi identiques" dans services/models.
- une seule source pour normaliser les tables incompatibles.

3.2 Couche repository metier (intermediaire entre service et model)
Creer:
- app/Repositories/BaseRepository.php
- app/Repositories/AcademicContextRepository.php
- app/Repositories/StudentRepository.php
- app/Repositories/InscriptionRepository.php
- app/Repositories/RapportRepository.php
- app/Repositories/SoutenanceRepository.php
- app/Repositories/AuditRepository.php

Role:
- encapsuler les requetes par aggregate metier (pas par page UI).
- supprimer la duplication cross-service.

Priorite technique:
- StudentRepository + InscriptionRepository d'abord
  (ce sont vos deux tables les plus transverses).

3.3 Couche service commune (transverse)
Creer:
- app/Services/Common/AcademicYearContextService.php
- app/Services/Common/AuditTrailService.php
- app/Services/Common/AuthorizationContextService.php
- app/Services/Common/DashboardMetricsService.php
- app/Services/Common/DocumentRenderService.php

Role:
- AcademicYearContextService:
  une seule regle pour recuperer/appliquer l'annee academique active.
- AuditTrailService:
  centralise logCreation/logModification/logValidation.
- DashboardMetricsService:
  factorise les KPI repetes.
- DocumentRenderService:
  factorise le tronc commun PDF/HTML->PDF.

3.4 Couche controller commune
Creer:
- app/controllers/BaseController.php
- app/controllers/traits/ActionDispatchTrait.php
- app/controllers/traits/ApiResponseTrait.php
- app/controllers/traits/FormValidationTrait.php

Role:
- BaseController:
  injection des dependances communes + helpers de reponse.
- ActionDispatchTrait:
  map action => methode (evite 50 if/elseif par controller).
- ApiResponseTrait:
  jsonSuccess/jsonError/redirect standardises.
- FormValidationTrait:
  validation uniforme des inputs GET/POST.

7) REDUIRE LE NOMBRE DE SERVICES/CONTROLLERS/MODELS
================================================================================

7.1 Controllers
Regle:
- 1 controller = orchestration HTTP uniquement.
- pas de SQL direct en controller.
- pas de logique metier lourde.

Action immediate:
- basculer les 5 controllers "a verifier" vers des services explicites.
- corriger les 4 references partielles de services introuvables.

7.2 Services
Regle:
- un service par capability metier.
- max de responsabilites claires (query, workflow, export, notif).
- utiliser Common services pour le transversal.

7.3 Models
Regle:
- models = acces DB atomique + mapping.
- pas de logique de workflow multi-objets.
- reutiliser BaseRepository/traits CRUD.

================================================================================
10) RESULTAT ATTENDU APRES CENTRALISATION
================================================================================

- Moins de fichiers touches par une evolution metier transversale.
- Moins de regressions SQL (une requete centralisee au lieu de N copies).
- Controllers plus lisibles et plus courts.
- Services decoupes par responsabilite.
- Models plus stables.
- Alignement plus fiable avec audit.txt.

En resume:
Le levier principal n'est pas de "fusionner en un seul bloc", mais de creer
des points de verite uniques (QueryCatalog, Repositories, Common Services,
BaseController, SchemaAliasResolver) et de faire converger progressivement
tout le code actuel vers ces points centraux.
